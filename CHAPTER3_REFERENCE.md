Executive Summary  
This chapter specifies a system architecture and design for a unified governance platform using Everything-as-Code (EaC). The architecture is layered, declarative, and continuously reconciled. It is designed around GitOps principles and Kubernetes control plane enforcement, such that the platform itself becomes the primary policy enforcement point, while CI remains an execution engine for building and verifying artifacts [1]-[4]. The proposed architecture uses Terraform for Infrastructure-as-Code (IaC), remote state and locking, Open Policy Agent (OPA) with Gatekeeper for Kubernetes admission control policies, Conftest for configuration policy testing in continuous integration, gosec for Go language static analysis, Trivy for container vulnerability scanning and SBOM workflows, Kaniko for daemonless image builds in Kubernetes, Tekton Pipelines for Pipeline-as-Code execution as Kubernetes custom resources, Tekton Chains for cryptographically signed provenance and attestations, and Argo CD for pull-based continuous delivery with continuous reconciliation and drift control [5]-[17]. Beyond the baseline toolchain, the chapter explicitly adds supply chain hardening for both pipeline code and application code, because unauthorized changes to pipeline definitions, manifests, or artifacts directly undermine governance. The added controls include protected branches, code owner review and required status checks in the version control system, commit and tag signing, immutable registry behavior, container image signature verification at admission time, SBOM generation and retention, Argo CD Projects and RBAC for multitenant isolation, admission controls and RBAC to prevent direct kubectl changes to production namespaces, and runtime monitoring that alerts on drift, audit violations, and policy denials [18]-[28]. The chapter provides tables to map tools to layers and to planes of responsibility, control plane, execution plane, evidence plane, then provides stepwise narratives for two change paths, firewall application change and governance change. It also includes a threat model that maps five required threats to preventive, detective, corrective controls, aligned with established secure development and container security guidance, and modern provenance and attestation frameworks such as SLSA, in toto, and Sigstore [29]-[36].

3.0 Introduction  
The transition toward highly distributed, dynamically orchestrated environments has fundamentally disrupted traditional software delivery paradigms, culminating in a pervasive governance crisis within modern computing [37]. As identified in the foundational literature reviewed in Chapter 2, engineering practices increasingly rely on a fragmented ecosystem of disparate tools, manual interventions, and isolated security checkpoints. This fragmentation creates severe operational vulnerabilities, including insidious configuration drift, inconsistent policy enforcement across multi-cloud environments, and escalating software supply chain threats that exploit weak validation boundaries [37]. To address these systemic risks, this chapter presents an exhaustive blueprint for a unified Everything-as-Code (EaC) architecture tailored specifically for cloud-native DevSecOps pipelines. This chapter serves strictly as the conceptual and architectural design of the proposed system, establishing the theoretical framework and structural interactions required to automate governance. It does not contain experimental results, performance numbers, or claims of observed improvements, as those elements are reserved for subsequent evaluation chapters.  
Chapter 3 defines the blueprint for how governance, security, and compliance are encoded, executed, and evidenced across the software delivery lifecycle for a cloud-native workload, a virtual firewall function. The virtual firewall is treated as a high-impact component, because it sits on critical traffic paths and often has elevated networking permissions compared with typical microservices. Therefore, the architecture must minimize trust in ad-hoc human action, minimize trust in external pipeline identities, and enforce deterministic controls at platform boundaries, especially admission and reconciliation [1], [3], [20], [26]. This chapter uses external primary sources to justify architectural mechanisms rather than specific vendor defaults. The cloud provider, Kubernetes distribution, cluster topology, and CI runner sizing are intentionally unspecified. Where a mechanism depends on a provider feature, such as registry tag immutability, the design describes the capability as an architectural requirement and cites one widely used reference implementation as an existence proof [28], [37]. The primary objective of this architecture is to relocate governance decisions from isolated silos and human gatekeepers directly into the foundational platform, thereby ensuring that security and compliance become continuous, deterministic, and verifiable properties of the software delivery lifecycle [37]. To contextualize and demonstrate the efficacy of this blueprint, the target workload navigating the proposed system is a cloud-native virtual firewall function, implemented as a containerized network function and deployed on a Kubernetes cluster [37]. Because this virtual firewall operates as a critical security enforcement mechanism, managing east-west and north-south network traffic while enforcing strict packet inspection rules, it must be treated as a highly sensitive, regulated workload [38].  
The architecture is therefore explicitly designed to protect two distinct domains. The first domain is the application code, encompassing the firewall source code, container images, Helm charts, Kustomize overlays, and deployment manifests. The second domain is the governance code itself, which includes continuous integration pipelines, policy repositories, infrastructure definitions, and continuous delivery configurations [37]. By securing both the workload and the underlying mechanisms that deliver it, the architecture establishes a zero-trust foundation for automated software delivery, guaranteeing that no unauthorized alterations can compromise the network perimeter.

3.1 Architectural Principles  
The proposed architecture is constructed upon five immutable design principles. These principles systematically dismantle the fragmentation and security risks identified in conventional DevSecOps methodologies, providing a rigorous theoretical foundation for the system design [37]. The architecture is defined by five principles: modularity, declarativity, immutability, observability, traceability. Each principle is presented with the problem addressed, the design decision forced, and the audit evidence produced.  
Modularity
Modularity is defined as the architectural decomposition of complex, monolithic governance requirements into independent, reusable, and logically isolated components [37]. Problem addressed: In complex cloud-native platforms, governance logic tends to fragment across teams and tools, which leads to inconsistent enforcement and unclear responsibility boundaries. A modular architecture decomposes the system into layers and components that can be evolved independently without rewriting the entire governance model. This aligns with Kubernetes extension patterns, where controllers and custom resources add capabilities as separable units, and with Tekton Pipelines where Tasks and Pipelines are reusable building blocks expressed as Kubernetes resources [12], [13]. In traditional environments, the deep entanglement of application logic, infrastructure configuration, and security policy means that a minor update to one domain often causes catastrophic failures in another. This principle directly addresses the problem of rigid, monolithic governance configurations that inhibit deployment velocity and create bottlenecks for engineering teams [37]. Design decision forced: Organize the architecture into four layers—infrastructure provisioning, policy enforcement, compliance verification, orchestration and delivery—and treat each tool as a module with a defined interface and trust boundary. For example, Terraform modules encapsulate infrastructure defaults, Gatekeeper constraint templates encapsulate admission logic, Tekton Tasks encapsulate CI steps, and Argo CD Applications encapsulate deployment intent [5], [9], [14], [17], [19]. Consequently, the architecture forces a design decision to mandate provider-agnostic modules for infrastructure provisioning and decoupled policy libraries for security enforcement. By isolating these domains, the architecture ensures that changes to a firewall network policy do not inadvertently disrupt the underlying compute infrastructure. Audit evidence: Modularization improves auditability because each module can emit specific evidence artifacts, such as Terraform plans and state history, Gatekeeper audit violation counts, Tekton Chains attestations, and Argo CD sync status transitions [25], [15], [3]. The evidence produced by this principle for auditability includes independent unit test reports for individual modules, logically isolated state files, and compartmentalized policy bundles that can be inspected independently by compliance officers.  
Declarativity
Declarativity refers to the practice of explicitly defining the desired end state of a system rather than scripting the imperative, sequential steps required to achieve that state [37]. Problem addressed: Imperative scripts and manual runbooks make enforcement outcomes dependent on operator context and timing, which undermines determinism and complicates evidence. GitOps explicitly requires that desired state be expressed declaratively, and Kubernetes itself is built around a declarative API where configuration describes the desired end state [1], [3]. The literature highlights that imperative operational models are highly susceptible to human error and opaque manual configurations, leading to untrackable system alterations and unpredictable deployment outcomes [37]. To mitigate this, the architecture forces the design decision that all infrastructure, policies, pipelines, and workloads must be codified using declarative configuration languages. This constraint eliminates procedural ambiguity, shifting the burden of execution logic entirely to the automated controllers. Design decision forced: Make all governance-relevant artifacts declarative, including Kubernetes manifests, Helm chart values, Kustomize overlays, Tekton Pipeline definitions, Gatekeeper constraints, and Argo CD Applications. Use declarative admission control extension points in Kubernetes to validate requests before persistence, ensuring that policy decisions are applied consistently [20], [21], [22], [26]. Audit evidence: Declarative artifacts are naturally versionable and can be tied to immutable identifiers, including Git commit hashes, container image digests, and cryptographically signed attestations. This allows an auditor to recreate the intended state and compare it to the actual state programmatically [37], [32], [34]. The tangible evidence produced by this principle is a collection of version-controlled manifests stored in a Git repository, which serve as the unequivocal, machine-readable source of truth for the entire environment. This ensures that any auditor can read the declared state and mathematically verify its compliance without needing to understand the underlying deployment scripts.  
Immutability
Immutability dictates that once a software artifact or infrastructure component is instantiated and validated, it is never modified in place [37]. Problem addressed: Configuration drift and silent changes are enabled when artifacts can be modified in place, such as mutable image tags, direct cluster edits, or retroactive pipeline changes. GitOps principles explicitly call for versioned and immutable desired state, and for agents that pull and reconcile that desired state continuously [1], [3]. The pervasive risk of configuration drift, where the live operational state diverges silently from the documented baseline due to manual hotfixes or unauthorized patches, is a primary failure mode in modern computing [37]. By adhering to immutability, the architecture forces the adoption of the single-writer principle, stipulating that only automated controllers possess the authority to alter the production cluster state [37]. Any required update necessitates the generation, validation, and deployment of an entirely new artifact, completely replacing the previous iteration. Design decision forced: Enforce immutability across three domains. First, desired state in Git is protected from unauthorized change through branch protections, review requirements, required status checks, and code owner controls. Second, artifacts such as container images and SBOMs are referenced by digest and protected by registry-level immutability settings and digital signatures. Third, production cluster state has a single logical writer, the GitOps controller, and direct changes are denied or reverted by reconciliation [18], [28], [2], [3]. This principle produces robust audit evidence in the form of read-only cryptographic digests, immutable container image tags, and a highly deterministic environment where unauthorized runtime changes are immediately overwritten by the reconciliation engine. Audit evidence: Immutability yields stable evidence chains. For example, OCI images are content-addressed by digest and include manifests and layers that are defined by specification, which supports artifact integrity verification when combined with signatures and attestations [37], [34].  
Observability
Observability is the capability to continuously infer the internal state and health of a highly distributed system based entirely on its external outputs, such as metrics, logs, and distributed traces [37]. Problem addressed: Automated governance without telemetry becomes brittle, because operators cannot distinguish between compliant convergence, policy blocks, and controller failures. Observability is required both for operational resilience and for compliance evidence collection. Prometheus defines a monitoring and alerting system centered on labeled time series and query-based alerts, while OpenTelemetry defines cross-signal correlation across traces, metrics, and logs via context propagation [23], [24]. Traditional security architectures suffer from a severe lack of visibility, resulting in silent failures, undetected policy violations, and shadow operations that bypass official governance channels [37]. To resolve this, the architecture forces the design decision to embed telemetry collection natively within the platform, utilizing a comprehensive suite of monitoring tools to capture every system event. This pervasive visibility ensures that the architecture continuously produces actionable evidence, emitting centralized logs and real-time metrics that feed directly back into the automated governance and self-healing reconciliation loops. By treating observability as code, the system guarantees that all compliance violations and drift events are recorded immutably, satisfying strict regulatory mandates. Design decision forced: Treat observability as an evidence plane with explicit data products, metrics, logs, traces, and audit events. Use Prometheus to scrape metrics and evaluate alerting rules, use centralized logging stacks such as Elastic Stack and Fluentd collection to retain logs beyond container lifetimes, and use OpenTelemetry to correlate signals for debugging and forensic investigation [23]-[27]. Audit evidence: Observability produces time-correlated evidence for both preventive and detective controls, including admission denial counts, audit run timestamps, GitOps sync status changes, and Kubernetes audit events, which are recorded in the API server and persisted by configured backends [25], [22].  
Traceability
Traceability is defined as the unbroken, verifiable, and cryptographically secure lineage of every system modification from its initial code commit to its final runtime execution [37]. Problem addressed: Governance collapses in incident response and audits if changes cannot be linked to an accountable origin. GitOps and secure development guidance both emphasize evidence and artifact integrity, and modern supply chain frameworks focus on provenance and attestation. NIST SSDF describes artifacts as evidence of secure practices and emphasizes securing artifacts and evidence handling, while in toto provides a framework for holistic supply chain integrity guarantees [29], [34]. The lack of a unified audit trail across fragmented continuous integration tools has historically rendered regulatory compliance an exhaustive, manual forensic exercise, severely delaying software releases [37]. To eliminate audit fatigue and mitigate sophisticated software supply chain threats, the architecture forces strict version control disciplines, mandatory peer reviews, and cryptographically signed release points. The tangible evidence produced by this principle includes the automated generation of software bill of materials (SBOM) documents, comprehensive provenance attestations, and a deterministic Git commit history that maps every production workload, including the virtual firewall, back to its exact source configuration and the specific human identity that authored the change [39]. Design decision forced: Assign stable identifiers to every lifecycle object and link them across the pipeline, Git commit identifiers, signed commits and tags, container digests, SBOM document identifiers, provenance attestations, Argo CD Application metadata, and Kubernetes audit events. Require that workload manifests carry traceability labels, and require that build outputs are signed and accompanied by provenance. Use Tekton Chains to observe TaskRuns and PipelineRuns and to generate cryptographically signed outputs, and use Sigstore tooling to verify signatures and attestations [15], [16], [35], [32]. Audit evidence: Traceability evidence includes commit signature verification status, signed container images and attestations, SBOMs in SPDX or CycloneDX formats, and reconciler logs and sync history. Kubernetes itself provides signed artifact verification guidance for its releases, which demonstrates that signature verification is an accepted operational approach in Kubernetes ecosystems [18], [30], [31].

3.2 Layered Architecture Design  
To operationalize the Everything-as-Code paradigm, the system is stratified into four interacting, logically separated layers. Figure 3.1, Layered reference architecture diagram, visually represents these four layers alongside their respective tooling, distinct trust boundaries, and explicit demarcations between the control plane and the execution plane. The virtual firewall function flows continuously through these layers, subjected to rigorous, automated validation at every trust boundary before reaching the runtime environment [37]. This section defines the four-layer reference architecture and maps each required tool to an architectural plane: control plane, execution plane, evidence plane. The key architectural move is to relocate governance decisions into the platform control plane, especially Kubernetes admission and GitOps reconciliation, while leaving build execution in the CI execution plane. This separation reduces the blast radius of compromised CI identities, because the pipeline can express intent by committing to Git, but the platform retains authority to apply changes [2], [3], [20].  
Control plane: Components that decide whether a change is allowed and what target state is authoritative, for example Kubernetes admission control, Argo CD reconciliation policies, and Argo CD Projects and RBAC settings [3], [6], [7], [20].  
Execution plane: Components that perform work that has already been authorized, for example Tekton Task Pods that build images, scanners that evaluate artifacts, and Terraform apply execution jobs. Tekton explicitly runs each Task as a Pod, which makes CI workloads first-class Kubernetes executions [12], [13].  
Evidence plane: Components and storage systems that retain telemetry and artifacts as evidence, for example Prometheus time series, Elastic Stack logs, OpenTelemetry traces and logs, Gatekeeper audit metrics, Kubernetes audit logs, SBOM documents, and cryptographic signatures and attestations for build outputs [22], [23], [24], [25], [30].

3.2.1 Layer 1: Infrastructure Provisioning, the Foundation
The first layer is responsible for the deterministic provisioning and lifecycle management of the foundational compute, network, and storage resources required to host the cloud-native workloads. This layer employs Terraform as the primary Infrastructure-as-Code engine [37]. The core responsibility of this layer is to ensure that the underlying environment, such as the Amazon Elastic Kubernetes Service cluster hosting the virtual firewall, is instantiated in a secure, repeatable, and fully documented manner. Baseline networking services, including virtual private clouds, isolated subnets, and identity access management roles, are codified extensively within this layer. A critical architectural boundary established in this layer is the abstraction of underlying cloud application programming interfaces through the use of provider-agnostic modules [37]. These modules encapsulate organizational best practices, ensuring that complex networking architectures and secure container registry services are deployed consistently across disparate environments. Furthermore, this layer mandates strict metadata tagging across all provisioned resources. These tags act as mandatory hooks for downstream policy enforcement, representing the critical linkage between infrastructure deployment and continuous compliance. By tagging resources with specific compliance scopes, the infrastructure layer enables downstream policy engines to dynamically ascertain which regulatory rules must be applied, ensuring that the virtual firewall network environment is governed by the correct regulatory framework [37].  
3.2.2 Layer 2: Policy Enforcement, the Gatekeeper
The second layer functions as the central, decoupled decision engine, systematically intercepting all proposed changes to ensure absolute conformity with organizational security mandates [37]. This layer utilizes the Open Policy Agent alongside its declarative query language, Rego, acting as the ultimate policy enforcement point. This layer enforces architectural guardrails at two distinct temporal boundaries. First, it performs static analysis of Terraform plans during the continuous integration pipeline, rejecting non-compliant infrastructure code before it can be merged into the main branch [37]. Second, it operates dynamically as a Kubernetes admission controller utilizing Gatekeeper, preventing non-compliant manifests from ever being scheduled onto the cluster. By utilizing the policy enforcement point concept, the architecture actively relocates governance decisions directly into the platform, drastically reducing the reliance on pipeline correctness [37]. If a pipeline is compromised, the platform itself will still reject malicious deployments. For the virtual firewall function, this layer evaluates at least three critical policy categories. First, it enforces an image registry allowlist and mandates image signature verification, ensuring the firewall container originates strictly from a trusted source and has not been tampered with [43]. Second, it requires specific metadata labels for traceability, ensuring that the workload owner, deployment environment, and compliance scope are explicitly declared within the deployment manifest. Third, it imposes stringent networking policy constraints specifically tailored for firewall pods. These constraints include privileged mode restrictions, strict host networking limitations, and policies that only allow precise kernel capabilities required for packet inspection, thereby preventing lateral movement and container escape vulnerabilities [44].  
3.2.3 Layer 3: Compliance and Security Verification, the Auditor
The third layer is dedicated to continuous verification, seamlessly integrating Compliance-as-Code (CaC) and Security-as-Code (SaC) directly into the developer workflow. This layer acts as an automated auditor, translating abstract regulatory frameworks into executable technical tests that run continuously against the codebase [37]. To maintain strict modularity, this layer employs a diverse suite of specialized scanning engines. Trivy is utilized for deep vulnerability scanning of the containerized firewall image, detecting known exposures in base layers and dependencies [45]. The gosec tool provides comprehensive static application security testing directly against the source code, identifying patterns of insecure coding practices [46]. Conftest is implemented to validate structured configuration files against the Rego policies defined in the second layer [47]. Furthermore, the architecture utilizes Kaniko for daemonless, secure container image building entirely within the user space, mitigating the risks associated with exposing Docker sockets within the continuous integration environment [48]. While tools such as Semgrep, SonarQube, and OWASP ZAP represent common industry standards for static and dynamic analysis, they are positioned within this architecture as optional, pluggable modules within the verification layer [37]. This strategic design choice ensures the architecture remains highly modular and adaptable to evolving threat landscapes. Crucially, all security findings generated in this layer are emitted as structured evidence. This evidence is fed directly back into the centralized observability layer, creating a continuous feedback loop that triggers automated alerts and influences subsequent deployment gates, perfectly aligning with the observability principle.  
3.2.4 Layer 4: Orchestration and Delivery, the Engine
The fourth layer serves as the operational engine of the architecture, coordinating the execution of the verification tools and managing the delivery of artifacts. This layer utilizes Tekton for continuous integration, defining pipeline tasks as native Kubernetes custom resource definitions, and Argo CD as the pull-based GitOps reconciliation controller for continuous delivery [37]. A paramount architectural principle enforced in this layer is the absolute separation of the execution plane from the control plane [37]. The continuous integration pipeline, powered by Tekton, operates solely within the execution plane. It is responsible for executing builds, running tests, and assembling artifacts. However, it possesses no authority to alter the production environment. Instead, continuous delivery authority resides strictly within the platform control plane managed by Argo CD. This layer explicitly enforces the single-writer principle [37]. Argo CD acts as the sole logical entity permitted to modify the production cluster state. It operates a continuous reconciliation loop, constantly comparing the live cluster state against the desired state declared in the Git repository. Any detected divergence is classified as drift. When drift occurs, Argo CD employs its strict prune and self-heal semantics to instantly revert the unauthorized changes, restoring the environment to its declared state [37]. By shifting the locus of control away from the pipeline and into the platform, this layer guarantees absolute state determinism and eliminates the possibility of unauthorized runtime modifications.

3.3 SCM-Centric Versus Platform-Centric CI Execution Models  
Understanding the precise placement of the locus of control requires a rigorous architectural comparison between execution models [37]. In distributed systems governance, the control plane is defined as the subsystem responsible for making authoritative decisions regarding what actions are permitted, scheduled, or rejected. Conversely, the execution plane is the subsystem that carries out the computational actions authorized by the control plane. The authority boundary represents the logical perimeter beyond which an actor or system is no longer authorized to dictate state changes. Finally, the policy enforcement point is the specific location within the architecture where governance rules are evaluated and applied [37].  
In source code management (SCM)-centric models, exemplified by traditional continuous integration engines such as Jenkins or GitHub Actions, the locus of control resides entirely within the pipeline itself. These legacy tools conflate the execution plane with the control plane. The pipeline is granted highly privileged credentials to actively push changes into the production cluster, creating a weak authority boundary that extends far outside the actual deployment environment. Because the policy enforcement point exists within the pipeline, any compromise to the pipeline identity immediately grants an attacker full administrative access to the cluster [49]. In contrast, the platform-centric model advocated by this architecture relocates the locus of control firmly within the Kubernetes platform. The continuous integration pipeline operates solely in the execution plane, building and verifying artifacts but committing only declarative manifests to Git. The control plane, managed by Argo CD, pulls these manifests and applies them only after rigorous admission validation. This design establishes a strong authority boundary at the cluster edge, where the policy enforcement point is embedded in the Kubernetes admission controller. Even if the pipeline is compromised, the platform retains ultimate veto power over deployments [49]. Table 2 compares the two models across key dimensions.

3.5 Change Flows for Application and Governance Code  
This architecture defines two distinct change paths to maintain separation between application and governance domains. Path A describes an application change to the virtual firewall code, visually detailed in Figure 3.2, Application change flow for firewall workload modifications. When a developer commits firewall code, the Git repository enforces protected branches, requiring pull request approval and status checks from scanners like gosec and Trivy. The Tekton pipeline executes in Kubernetes Pods, building with Kaniko, scanning with Trivy (generating SBOM), and testing policies with Conftest. Successful runs produce signed images and attestations via Tekton Chains. The commit merges to main, triggering Argo CD to detect the change. Argo CD reconciles by applying manifests after Gatekeeper admission validates against OPA policies (e.g., image signature, labels, networking constraints). Once admitted, the workload runs, and observability tools emit logs, metrics, and traces. Alerts open immediately if drift or policy violations occur. In this path, the Git repository is the absolute source of truth for manifests, while the artifact registry serves as the source of truth for the binary [37].  
Path B describes a governance change, encompassing pipeline code, policy code, or Terraform module modifications, visually detailed in Figure 3.3, Governance change flow for policy and pipeline modifications. When a security engineer commits a modification to the Rego policies, the repository enforces strict CODEOWNERS approval, ensuring that only designated security personnel can authorize the change [50]. The continuous integration pipeline performs validation of the policies using Conftest tests. Upon successful validation, the commit results in a signed release. The policy bundles undergo a controlled rollout to the cluster, managed by Argo CD. The observability layer provides continuous monitoring for policy denials and controller changes, ensuring the new governance rules do not disrupt legitimate network traffic. For Path B, the policy repository serves as the absolute source of truth for organizational governance [37].

3.7 Toolchain Rationale and Design Alternatives  
The selection of specific tools within this architecture is entirely driven by the necessity to fulfill the five architectural principles and comprehensively address the severe governance fragmentation and tool sprawl identified in Chapter 2.  
Terraform is selected as the primary Infrastructure-as-Code engine because its robust state management, extensive provider ecosystem, and modular architecture enable the declarative provisioning of complex, multi-environment infrastructures [37]. While alternatives like CloudFormation are tightly coupled to a single vendor, and configuration managers like Ansible are fundamentally procedural, Terraform supports the provider-agnostic modularity required by the architecture. Pulumi presents a viable context by utilizing general-purpose programming languages, but Terraform and its configuration language enforce a stricter separation between data and logic, significantly reducing the complexity of auditing infrastructure states for compliance officers [55].  
Open Policy Agent, utilizing Rego and Gatekeeper, is chosen as the universal policy engine due to its profound ability to decouple policy logic from application code and enforce rules deterministically [37]. While Kyverno serves as a highly capable Kubernetes-native admission controller, the Open Policy Agent provides a standardized, domain-agnostic language that can simultaneously evaluate Terraform plans during continuous integration and enforce admission control during deployment, fulfilling the Everything-as-Code mandate for unified governance.  
Tekton is selected over traditional tools like Jenkins because it defines continuous integration pipelines natively as Kubernetes custom resource definitions [37]. Jenkins relies heavily on an imperative, plugin-heavy architecture that struggles to maintain immutability and fundamentally violates the separation of the control and execution planes. Tekton treats pipelines as declarative code, integrating seamlessly with the platform-centric execution model and natively supporting advanced supply chain security features.  
Argo CD is chosen as the continuous delivery controller because it is purpose-built for pull-based GitOps reconciliation [37]. While Flux is a comparable and highly effective GitOps controller, Argo CD provides superior multi-tenant security boundaries through its AppProject custom resources and granular role-based access control [53]. These precise controller-level controls are critical when governing sensitive workloads like the virtual firewall across diverse cluster environments, ensuring strict isolation and access management.

3.8 Expected Benefits as Architectural Properties  
By strictly adhering to the Everything-as-Code paradigm and strategically relocating the locus of control to the platform boundary, the proposed architecture is expected to manifest several critical operational properties without requiring claims of measured experimental improvement.  
First, the architecture guarantees absolute traceability. Because every infrastructure configuration, security policy, pipeline definition, and application manifest is version-controlled, peer-reviewed, and cryptographically signed, organizations possess a deterministic audit trail [37]. This trail maps every runtime behavior to a specific, approved code commit, ensuring total accountability. Second, the design drastically reduces the risk of configuration drift. The continuous reconciliation loop operated by the single writer ensures that manual, out-of-band changes are autonomously detected and overwritten, maintaining the pristine integrity of the virtual firewall deployment [37].  
Third, the system generates continuous compliance evidence natively. Rather than relying on periodic, manual sampling by human auditors, the architecture inherently produces real-time attestations, software bill of materials, and rich observability telemetry. This pervasive data collection significantly accelerates audit readiness and reduces compliance overhead. Finally, the architecture ensures profound supply chain hardening. By enforcing artifact immutability, mandatory signature verification, and a pull-based delivery model that minimizes external trust, the architecture systematically mitigates the attack vectors commonly exploited in modern software supply chain compromises.

3.9 Summary  
Chapter 3 has established the comprehensive architectural blueprint required to execute unified Everything-as-Code governance within cloud-native DevSecOps pipelines. By synthesizing declarative infrastructure provisioning, decoupled platform-centric policy enforcement, continuous security verification, and pull-based GitOps reconciliation, the design systematically mitigates the fragmentation, configuration drift, and supply chain threats identified in the foundational literature. With the theoretical boundaries, trust models, threat mitigations, and control flows firmly defined for the virtual firewall workload, the following chapter will detail the concrete implementation and procedural configuration required to realize this architecture in practice.
